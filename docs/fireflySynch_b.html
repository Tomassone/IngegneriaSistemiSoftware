fireflySynch

Introduction

Goal: Costruire il simulatore di un sistema costituito da 3 lucciole. .

Requirements

Costruire il simulatore di un sistema costituito da 3 lucciole. Inizialmente, le lucciole lampeggiamo ciascuna con una frequenza (costante) propria.

  • B) Il sistema è dotato anche di un sonar. Quando il sonar rileva un ostacolo a una distanza minore di una distanza prefissata DMIN, le lucciole devoono lampeggiare in modo sincronizzato. Quando la distanza dell’ostacolo (ri)supera DMIN, le lucciole tornano a lampeggiare in modo random.

Requirement analysis

Dall'analisi dei requisiti, emerge per il sistema la necessità di 3 entità distinte, cioè 3 lucciole. Esse verranno modellate come QActor firefly, data la loro natura allo stesso tempo reattiva (capacità di reagire a stimoli esterni) ed attiva (necessità di lampeggiare autonomamente): 

			Context ctxfirefly ip [ host="localhost" port=8040 ]

			QActor firefly context ctxfirefly{
				State s0 initial{
				/* inizializzazione stato lucciola per lampeggio
				   autonomo e successiva sincronizzazione */
				}
			}

In secondo luogo, il sistema si comporrà anche di un'entità sonar (pure in questo caso in forma del QActor sonar), che comunicherà periodicamente alle lucciole la presenza o meno di un ostacolo con distanza minore di quella prefissata DMIN: 

			QActor sonar context ctxfirefly {
				State s0 initial {
					/* inizializzazione stato sonar */
				}
			}

Problem analysis

Anche in questo caso, si ricorrererà ad un'entità che si occuperò della visualizzazione a schermo dello stato delle entità lucciole. Essa verrà modellato dal QActor griddisplay, dato come già realizzato.

Si definirà dunque l'architettura del sistema fireflySynch. Avremo due contesti distinti: ctxfirefly (per le firefly e per l'sonar) e ctxgrid (per griddisplay).

Gli attori precedentemente elencati potranno infine essere strutturati nel modo seguente (con l'aggiunta di un ulteriore attore "di utilità" creator, per l'istanziazione agevole dei 3 attori firefly richiesti): 

			QActor creator context ctxfirefly{
				State s0 initial{
					/* istanziazione 3 attori firefly */
				}
			}

			QActor firefly context ctxfirefly{
				State s0 initial{
				/* inizializzazione stato lucciola */
				}
				Goto rand
				State rand{
					/* randomizzazione frequenza lucciole */
				}
				Goto flash
				State flash{
					/* cambio di stato acceso/spento -> comunicato in dispatch all'attore
					griddisplay (descritto sotto)*/
				}
				Transition t0
				whenTimeVar Freq -> rand /* flash ripete ogni Freq secondi (ma se sync
				                            non viene ricevuto, la frequenza viene randomizzata) */
				whenEvent syncevent -> sync /* cambio di stato dopo l'arrivo 
				dell'evento sync (descritto sotto)*/

				State sync{
					/* sincronizzazione frequenza lucciole */
				}
				Goto flash
			}

Da notare che il passaggio della sincronizzazione avverrà tramite l'emissione dell'evento syncevent da parte dell'sonar: esso verrà ricevuto dalle firefly, che aggiorneranno la propria frequenza di lampeggio ad un valore comune espresso da una variabile nascosta nel file sorgente .qak delle lucciole. Si suppone che la comunicazione con il componente hardware avvenga tramite eventi, anche se in linea teorica potrebbe anche avvenire tramite polling (piuttosto inusuale). 

			QActor sonar context ctxfirefly {
				State s0 initial {
					/* inizializzazione stato sonar */
				}
				Goto getdist

				State getdist{
					/* polling del valore rilevato dal sonar */
				}
				Transition t0
				Goto syncstate /* se la distanza è minore di DMIN */

				State syncstate{
					/* invio di evento syncevent alle firefly */
				}
				Goto getdist
			}

Per semplicità, si supporrà che i diversi attori eseguano in un contesto locale (dunque sullo stesso computer), anche se chiaramente sarebbe possibile per il sistema funzionare nel distribuito.

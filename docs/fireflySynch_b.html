fireflySynch

Introduction

Goal: Costruire il simulatore di un sistema costituito da 3 lucciole. .

Requirements

  • Il sistema deve essere costituito da tre lucciole indipendenti.
  • Inizialmente, le lucciole lampeggiamo ciascuna con una frequenza (costante) propria.
  • Il sistema deve anche essere dotato di un sonar.
  • Quando il sonar rileva un ostacolo a una distanza minore di una distanza prefissata DMIN, le lucciole devono lampeggiare in modo sincronizzato.
  • Quando la distanza dell’ostacolo (ri)supera DMIN, le lucciole tornano a lampeggiare in modo random.

Requirement analysis

Dall'analisi dei requisiti, emerge per il sistema la necessità di 3 entità distinte, cioè 3 lucciole. Esse verranno modellate come QActor firefly, data la loro natura allo stesso tempo reattiva (capacità di reagire a stimoli esterni) ed attiva (necessità di lampeggiare autonomamente).

In secondo luogo, il sistema si comporrà anche di un'entità orchestratrice (pure in questo caso in forma del QActor orchestrator), che riceverà le informazioni captate dal sonar e che comunicherà dunque alle lucciole di sincronizzarsi o di randomizzare la propria frequenza di lampeggio, a seconda dei casi.

Infine, si ricorrererà ad un'entità che si occuperò della visualizzazione a schermo dello stato delle entità lucciole. Essa verrà modellato dal QActor griddisplay, dato come già realizzato.

Problem analysis

Come primo passo, si definirà l'architettura del sistema fireflySynch. Avremo due contesti distinti: ctxfirefly (per le firefly e per l'orchestrator) e ctxgrid (per griddisplay).

Gli attori precedentemente elencati potranno infine essere strutturati nel modo seguente (con l'aggiunta di un ulteriore attore "di utilità" creator, per l'istanziazione agevole dei 3 attori firefly richiesti): 

			QActor creator context ctxfirefly{
				State s0 initial{
					/* istanziazione 3 attori firefly */
				}
			}

			QActor firefly context ctxfirefly{
				State s0 initial{
				/* inizializzazione stato lucciola (frequenza randomizzata) */
				}
				Goto flash
				State flash{
					/* cambio di stato acceso/spento -> comunicato in dispatch all'attore
					griddisplay (descritto sotto)*/
				}
				Transition t0
				whenTimeVar Freq -> flash /* flash ripete ogni Freq secondi */
				whenEvent syncevent -> sync /* cambio di stato dopo l'arrivo 
				dell'evento sync (descritto sotto)*/
				whenEvent randevent -> rand /* cambio di stato dopo l'arrivo 
				dell'evento rand (descritto sotto)*/

				State sync{
					/* sincronizzazione frequenza lucciole */
				}
				Goto flash

				State rand{
					/* randomizzazione frequenza lucciole */
				}
				Goto flash
			}

Da notare che il passaggio della sincronizzazione avverrà tramite l'emissione dell'evento syncevent da parte dell'orchestrator: esso verrà ricevuto dalle firefly, che aggiorneranno la propria frequenza di lampeggio ad un valore comune espresso da una variabile nascosta nel file sorgente .qak delle lucciole. Viceversa, il passaggio alla randomizzazione verrà segnalato dall'evento randevent. Si suppone che la comunicazione con il componente hardware avvenga tramite polling per semplicità. In alternativa, si potrebbe anche definire un ulteriore attore la cui funzione sia quella da fare solamente da wrapper per il sonar (non necessario per un'architettura così semplice). 

			QActor orchestrator context ctxfirefly {
				State s0 initial {
					/* inizializzazione stato orchestratore */
				}
				Goto getdist

				State getdist{
					/* polling del valore rilevato dal sonar */
				}
				Transition t0
				Goto syncstate /* se la distanza è minore di DMIN */
				Goto randstate /* se la distanza è maggiore o uguale a DMIN */

				State syncstate{
					/* invio di evento syncevent alle firefly */
				}
				Goto getdist

				State randstate{
					/* invio di evento randevent alle firefly */
				}
				Goto getdist
			}

Per semplicità, si supporrà che i diversi attori eseguano in un contesto locale (dunque sullo stesso computer), anche se chiaramente sarebbe possibile per il sistema funzionare nel distribuito.
